ENGINEERING_RULES

本文档为工程的工程契约与开发规范边界。所有 Agent 行为与开发工作必须将其作为强制基准执行。

一、规划与开发工作流规则
	1.	强制前置规划：必须执行 Research 与 Plan 并在得到确认后，方可进入 Implement。
	2.	里程碑严格隔离：当前专注当前阶段（如 MVP），功能点不可跨阶段泄露；“以后再做”的功能必须抛入 Backlog。
	3.	基于 Issue 驱动：每个功能或修复必须以确定的目标/Issue 为原点，明确验收标准与边界。
	4.	原型重构容忍：原型阶段允许为保证当前阶段质量而推翻重来，代码成本低于方向错误成本。
	5.	JSON 生成严谨性：生成的 JSON 必须通过严格校验（无遗留注释、无尾随逗号），且严禁改变预先约定的 Schema 结构。

二、版本控制与并行规范
	1.	分支管理：采用 main + feature branches 模型。禁止向 main 分支直接 Push。
	2.	单域隔离：每个功能/Issue 必须使用独立的本地分支，避免变更污染。
	3.	并行开发防护：多任务并行时必须采用独立 Worktree；Agent 在独立会话中仅处理职责内的 Issue。
	4.	合并前置卡点：所有 PR 或合并操作，必须完成冲突检查与主线回归验证。
	5.	清晰提交：Commit Message 必须能够显式关联到所处理的目标或 Issue。

三、环境与命令执行规范
	1.	开发模式优先：涉及 UI 与 Web 开发时强制使用开发服务器（如 npm run dev）保留热更新能力。
	2.	禁止本地生产构建：开发会话交互期间严禁运行破坏开发缓存或状态的构建命令（如 npm run build）。
	3.	依赖同步刷新：只要有包管理器锁定文件（Lockfile）变更，必须重启开发服务器应用更改。
	4.	命令自动化封装：强制使用预定义的自动化动作（如 update-docs、retro）来处理常规文档与复盘操作。

四、文档维护与状态同步规则
	1.	及时状态记录：完成任意功能后自动触发 docs/status.md 和 docs/changelog.md 记录。
	2.	架构特征同步：牵涉到模块或业务逻辑更改，务必更新 docs/architecture.md 及对应 feature-notes。
	3.	不可变文件保护：任何开发与文档记录行为，都不得修改工程定义的“不可变源文件”（如 JSON 模版库）。

五、DoD 验收卡点标准
	1.	规划一致性：最终实现未偏离 Plan/Spec 限定的范围。
	2.	运行可验证：交付了可执行的测试、脚本或明确的手工验收步骤，且已通过。
	3.	上下文已刷新：涉及的所有状态、日志及架构文档已同步写回完毕。
	4.	复盘已沉淀：遇到的重要错误已抽象为防止复发的规则存入相应的记忆库。
	5.	集成无回归：已保证基线无冲突，上下游基础链路保持畅通。

EXECUTION_GUARDRAILS

本文档定义了开发执行期的硬性边界、操作红线与标准卡点流程。触发红线行为将被视为严重偏差。

一、行为红线 (Prohibited Actions)
	1.	禁越过规划：严禁无 Research 与 Plan 前置输出而直接进行代码层面的 Implement 操作。
	2.	禁修改模板：禁止覆盖、修改或回写 Template_Classification_JSON 等被界定为“只读”的源数据文件。
	3.	禁主线直推：严禁利用任何手段绕过合并审核机制，直接向 main 分支进行推送代码。
	4.	禁范围蔓延：严格禁止脑补未在当前阶段（如 MVP）规定的需求，或自行将后期规划（V2/V3）提前夹带。
	5.	禁重命名基建：严禁重命名已有的核心基准文件、预定义的目录结构或既定术语。

二、变更卡点流程 (Change Workflow)

任何涉及功能的新增或修复，强制进入四步卡点流水线（B1 规则抽象）：
	1.	Research (研究前置)：遇新概念、新 API，必须输出精简研究笔记并验证其可行性。
	2.	Plan (计划锁定)：明确功能实现的步骤拆解与验收点，未经认可不可推进。
	3.	Implement (代码执行)：约束代码修改只发生在 Plan 范围内的时间和文件边界上。
	4.	Test & Sync (测试与同步)：验证运行无误后，立即执行 update-docs 流，阻断变更上下文丢失风险。

三、危险操作黑名单 (Dangerous Operations)
	1.	环境破坏命令：在 Agent 持久交互中执行 npm run build 或类似阻断本地调试的命令。
	2.	记忆池污染：无脑将海量代码细节或长篇对话完整灌入 PROJECT.md；禁止其成为流水账。
	3.	掩饰性补丁：修 Bug 仅在代码层做补遗（Patch），而拒绝执行 retro 将错误根因沉淀为新规则的心智惰性。

PROJECT_MEMORY_POLICY

本工程将上下文保存在明确划分目的的持久化文档中，旨在阻止 AI 记忆膨胀或逻辑漂移。

一、索引与事实源读取规则
	1.	入场必读：每次会话切入必须优先读取 PROJECT.md，以此作为全局路标与约束入口。
	2.	事实采信优先级：
	•	最高：官方文档
	•	次高：规范文件（如 AGENTS.md、系统约束文档、不可变数据模板等）
	•	基准：PRD / SKILL / 规格说明文档
	3.	按需下钻读取：
	•	拒绝对全体文档进行无效的全量读取。对于系统结构下钻查询 docs/architecture.md。对于进度查询查阅 docs/status.md。

二、文档更新与抗膨胀机制
	1.	只记增量边界：向 PROJECT.md 中新增的工程约束必须是“一句可执行的硬规则”，并放置详尽内容的外链。拒绝长篇大论。
	2.	根因沉淀：当发生严重错误或逻辑偏差时，除了修改代码本身，强制由 retro（复盘）抽象出一条防呆规范沉淀至项目知识库。
	3.	流式记录：不依赖 Agent 自身短时记忆。所有重要上下文在当前任务完成后，必须通过写回机制即刻转存到硬盘文档中。

三、术语与主线防漂移机制
	1.	核心阶段冻结：当前版本的 Milestone（例如 MVP 主闭环路径）落盘后即视作绝对主线。突发的新需求或想法全部阻塞并入 Backlog。
	2.	术语全量锁定：文档、源码、对话中的专有概念（如 Core、Work、某具体业务系统名称等）定义全局固化。
	3.	禁止口语替换：严禁代码注释或日志运用近似词、私造缩写顶替既定术语。若有新增概念，需在架构文档显式注册后统一应用。
