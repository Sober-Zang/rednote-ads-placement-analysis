# Eval Plan

状态：Phase 1 已锁定  
用途：定义本项目首轮评估思路、主检验样本、补充边界场景、验收标准与何时生成正式 `evals/evals.json`。

## 1. 评估目标

- 检查未来 Skill 是否能从混合文本中正确提取全部小红书链接
- 检查是否能完整下载所有有效小红书链接对应的图文、评论、互动信息与辅助材料
- 检查是否能将每个链接的数据合理分目录存储，并把下载结果正确传递给分析阶段
- 检查是否能依据 Prompt 生成符合 `Original_Prompt.md` 要求的单篇与综合报告
- 检查是否能把过程文件、Prompt 文件、报告文件完整落档
- 检查是否能正确排除非小红书链接并在最终播报中点名说明

## 2. 主检验样本

本轮主检验样本使用以下混合输入，包含 5 个小红书链接和 1 个 GitHub 链接：

```text
旧手机到底怎么卖才不亏？ 之前处理旧手机真的踩坑踩... http://xhslink.com/o/3WzkCVpHEfV
复制后打开【小红书】查看笔记！ 求助！旧手机想卖高价，到底该选啥平台 我嫌弃之前的... http://xhslink.com/o/ArxRxhOEdWf
复制后打开【小红书】查看笔记！ 回收平台水太深？热门渠道避雷硬核测评来喽 宝子们，... http://xhslink.com/o/7Jllb5XdsPy
复制后打开【小红书】查看笔记！ 重庆小白回收手机，还能这样？ 我最近想把家里的苹果1... http://xhslink.com/o/7Ows0hAttd6
复制后打开【小红书】查看笔记！ 近年iPad热门出手机型来了！看看转转回收价 哈喽大家... http://xhslink.com/o/8TOk0VCFYd6
复制后打开【小红书】查看笔记！ https://github.com/ssocean/PPT_Pilot
```

### 2.1 主样本要验证的事情

- 是否正确提取 6 个候选链接
- 是否识别其中 5 个为小红书链接，1 个为 GitHub 非目标链接
- 是否为 5 个有效小红书链接逐一下载并合理整理以下材料：
  - 图文正文文本
  - 图片
  - 评论区内容
  - 互动数据
  - 标题、作者、标签、发布时间等辅助信息
  - 评论词云图等衍生资产（如能力可用）
- 是否为每个有效链接创建独立样本目录，并按“完整标题 + 作者昵称”的规则落盘管理
- 是否将下载得到的原始材料正确传递给分析阶段，而不是停留在下载层
- 是否对 5 个有效样本分别生成单篇分析报告
- 是否基于这 5 份单篇报告生成 1 份综合分析报告
- 是否将 Prompt、manifest、原始输入、下载材料、单篇报告、综合报告全部落档
- 是否在最终播报中明确指出 GitHub 链接未纳入分析

### 2.2 主样本通过标准

- 5 个小红书链接全部被识别并进入下载流程
- GitHub 链接被剔除，不进入下载与分析
- 每个有效链接的样本目录中至少有可用于分析的文本、图片、评论和可用互动数据
- 每个样本生成 1 份单篇分析报告
- run 中生成 1 份综合分析报告
- 所有关键结论带引用标签并能追溯到实际文件
- 最终播报给出输出目录、报告路径和被排除链接说明

## 3. 补充边界场景

### 场景 A：有效小红书链接数为 0

目标：

- 验证无有效小红书链接时的错误返回
- 验证不会误进入下载和分析流程

```text
请分析这几个链接：

https://www.example.com/demo
https://github.com/ssocean/PPT_Pilot
```

通过标准：

- 返回明确错误提示
- 不创建分析报告
- 不把非小红书链接误判为有效样本

### 场景 B：用户给出额外提示词，必须走 run 专属 Prompt

目标：

- 验证 Prompt 分流逻辑
- 验证 run 专属 Prompt 不写入全局 assets

示例 Prompt：

```text
请分析以下小红书图文链接，但这次我不需要常规投放方法论，我只想重点比较它们在“评论区如何降低用户决策阻力”上的共同点，并把报告结构改成“结论-证据-风险”。

http://xhslink.com/o/3WzkCVpHEfV
http://xhslink.com/o/ArxRxhOEdWf
```

通过标准：

- 不使用标准 Prompt 资产作为本次 run 的实际 Prompt
- 在当前 run 中生成专属 Prompt 文件
- 专属 Prompt 只服务当前 run，不进入全局 `assets/`

## 4. 首轮验收标准

### 完整性

- 能从自然语言中抽出全部候选链接
- 能识别至少 1 个有效小红书链接时继续流程
- 有效链接为 0 时给出明确错误反馈
- 能为每个有效链接下载并整理完整分析材料
- 能将下载材料传递到分析阶段而不是只停留在抓取阶段

### 正确性

- 非小红书链接被剔除且被点名说明
- 标准 Prompt 与 run 专属 Prompt 的判定符合既定规则
- 多链接时综合报告基于本次 run 的全部单篇报告
- 单篇报告与综合报告符合 `Original_Prompt.md` 中的分析框架、输出格式与引用要求

### 落盘一致性

- 输出目录、文件命名、目录结构符合 `docs/run-blueprint.md`
- 每个链接都有独立样本目录
- 下载原始数据、Prompt、单篇报告和综合报告都生成到正确位置
- 关键结论带引用标签，并能追溯到 `reference_index.json`

### 边界控制

- run 专属 Prompt 没有写入全局 `assets/`
- 没有把非小红书链接纳入分析结果
- 没有把历史 run 的数据混入当前 run

## 5. 定性检查项

- 单篇报告不是泛泛总结，而是围绕“为什么有效、如何有效、是否可迁移”展开
- 综合报告不是重复单篇内容，而是提炼共性机制
- 报告里真正使用了文本、图片、评论、互动信息及其关联分析
- 引用来源写法足够支撑复核，不只是空标签
- 最终播报便于用户快速定位输出文件

## 6. 何时生成正式 `evals/evals.json`

当前阶段不生成，原因如下：

- 真实 skill 目录尚未创建
- 正式 `SKILL.md` 尚未定稿
- 代码与目录实现尚未开始，过早生成会导致路径与断言失真

下一阶段满足以下条件后再生成：

- 已创建真实 skill 目录
- 正式 `SKILL.md` 已定稿
- run 目录与输出路径已在代码层可执行
- 至少能跑通一条最小样例闭环

## 7. 与 `skill-creator` 流程的衔接

- 当前文件相当于 `skill-creator` 中的测试 Prompt 预案
- 等真实 skill 成型后，再把这里的主样本和补充场景转成正式 `evals/evals.json`
- 是否添加 assertions，应以“能客观验证”为前提，不强行形式化主观分析质量
- 当前主检验标准优先看端到端任务目标是否达成，而不是形式化 benchmark 分数

## 8. 本文件的验收条件

- 评估目标覆盖输入、下载、传递、分析、落档和播报六个方面
- 主检验样本已切换为用户指定的 6 段混合输入
- 补充场景已覆盖“0 有效链接”和“run 专属 Prompt”两类边界
- 清楚说明当前为什么还不生成正式 `evals/evals.json`
