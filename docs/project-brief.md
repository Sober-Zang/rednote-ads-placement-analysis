# Project Brief

状态：Phase 1 已锁定

## 0. 北极星与定位

- 不可变对齐源：`RAPA_Support_Documents/Original_Prompt.md`
- 项目定位：面向转转广告分析场景的内部研究原型
- 交付形态：一个由 `SKILL.md` 驱动的通用 Agent Skill
- 本阶段性质：仅做文档锁定，不做代码、脚手架接入或上游仓库改造

## 1. 项目目标

- 构建一个面向小红书图文广告投放分析的 Agent Skill，用于处理用户一次上传的多个小红书链接。
- 通过复用 `MediaCrawler` 的小红书数据获取能力与 `video-copy-analyzer` 的 Skill 组织方式，完成“链接输入 -> 内容采集 -> 单篇分析 -> 跨样本共性总结 -> 方法论文档沉淀”的完整闭环设计。
- 将过程数据、引用证据和最终结论全部落盘为本地文档资产，供用户核查。

## 2. 目标用户

- 转转内部广告投放分析专员
- 内容策略、素材策略、投放方法论沉淀相关成员
- 需要复核分析依据、查看原始证据与输出报告的内部研究使用者

## 3. 核心问题

- 用户当前缺少一个稳定的、可复查的流程，把多个小红书图文链接的内容和评论拆解为“为什么高 ROI / 高 CVR”的结构化判断。
- 现有人工分析难以同时兼顾逐篇深度、跨样本共性、引用溯源和本地归档。
- 项目需要在尽量复用成熟上游项目的前提下，避免从零搭建整套抓取与分析体系。

## 4. MVP 必须功能

- 接收用户自然语言中的多个链接，并从混杂文本中提取 `http://xhslink.com/` 小红书链接。
- 能识别并剔除非小红书链接，并在最终播报中明确说明未纳入分析的链接。
- 围绕“用户上传链接对应的帖子明细抓取”形成第一条闭环：每个链接一个文件夹，保留图片、文本、评论、互动数据与引用锚点。
- 基于标准分析 Prompt 或 run 内专属 Prompt，生成单篇分析报告。
- 当有效链接数大于 1 时，基于本次 run 的全部单篇报告生成综合分析报告。
- 将所有下载内容、过程文件、提示词文件、分析结果统一落到本地 run 目录中。
- `SKILL.md` 的结构、文风、流程组织与 `video-copy-analyzer` 强对齐，只在任务相关位置做最小必要改动。

## 5. 可砍功能

- 不属于首条闭环的 Web UI、可视化界面、数据库持久化
- 面向外部用户的分发、鉴权、支付、云部署
- 非小红书平台适配
- 对 `MediaCrawler` 全量能力的立即暴露式封装
- 自动化 benchmark 运行器与批量评测流水线的实现代码

## 6. 成功标准

- 文档层：`Project Brief`、`Project Spec`、`skill-draft`、`run-blueprint`、`eval-plan` 五类核心文档齐全，且无阻塞实现的歧义。
- 产品层：下一阶段实现者无需自行决定输入输出接口、run 命名、提示词分流规则、报告命名和上游复用基准。
- 约束层：`Original_Prompt.md` 未被修改；本阶段无代码、脚本、依赖或上游仓库接入动作。
- 对齐层：所有硬要求都能在文档中找到明确落点。

## 7. 约束

- `Original_Prompt.md` 视为不可变北极星文档。
- `skill-creator` 在本项目中默认保持启用，用于辅助 Skill 创建、文档设计和后续评估规划。
- 当前项目只按“内部研究原型”定位，不承诺当前阶段满足商用合规要求。
- `MediaCrawler` 当前只作为研究型参考基座；其商业化可用性需在下一阶段单独审查。
- 文档默认中文为主；技术名词允许保留英文。
- 标准 Prompt 仅在“无额外要求”或“用户目标与标准任务严格等价”时复用，否则必须写入 run 专属 Prompt 文件。
- 所有重要输出必须落到本地文件系统，不能只停留在对话里。

## 8. 明确不做

- 本阶段不创建最终 `SKILL.md`
- 本阶段不创建真实 skill 目录、脚本、数据抓取代码或分析代码
- 本阶段不接入 `MediaCrawler` 或 `video-copy-analyzer` 上游仓库
- 本阶段不修改 `Original_Prompt.md`
- 本阶段不把 V2/V3 的扩展能力混入 MVP 实现范围
